## 1. GGUF란

**GGUF (Georgi Gerganov Unified Format)**
- 딥러닝 모델(주로 LLM)을 효율적으로 저장·배포하기 위한 파일 포맷
- 주로 LLM 추론에 사용되며, **GGML** 라이브러리 기반의 런타임(`llama.cpp` 등)에서 읽도록 설계됨

한마디로 정리하면 **"LLM계의 Docker"** — 모델 가중치와 실행에 필요한 메타데이터를 하나의 파일에 담아, 어디서든 그대로 불러와 실행할 수 있게 해준다.

---

## 2. 탄생 배경

초기 LLM 모델은 보통 PyTorch 포맷인 `.pt`로 저장되었다.

- 훈련(학습) 목적으로는 문제가 없다.
- 하지만 모델을 **추론하고 서빙**하는 데는 한계가 명확했다 (효율성/접근성 부족).
- 이를 극복하기 위해 **GGML**이라는, CPU에서도 행렬 연산을 효율적으로 수행하기 위한 경량 텐서 라이브러리가 나왔다.
- 그러나 GGML만으로는 유연성/호환성이 떨어지고 설정을 수동으로 맞춰야 하는 문제가 있었다 → 새로운 파일 포맷이 필요했다.

그래서 Georgi Gerganov가 새로 개발한 것이 딥러닝 모델 저장용 단일 파일 포맷, **GGUF**다.

---

## 3. GGUF의 특징

- **단일 파일 구조** — 모델을 하나의 파일로 쉽게 공유할 수 있다.
- **메타데이터와 텐서 데이터 포함** — 모델의 가중치 텐서 값들과 메타데이터가 key-value 형식으로 함께 저장된다.
- **다양한 양자화 지원** — FP16뿐 아니라 양자화된 8, 6, 5, 4, 3, 2bit까지 다양한 텐서 타입을 지원한다.
- **호환성** — GGML 라이브러리 기반의 런타임에서 주로 사용되고, GPU 없이 CPU에서도 실행 가능하다.
- **확장성** — 새로운 기능을 추가해도 기존 모델과의 호환성이 유지된다.

---

## 4. GGUF 파일의 구조

GGUF 파일은 아래 순서로 저장된다.

**① 헤더**

```c
uint32_t magic;      // "GGUF" 문자열의 매직 넘버
uint32_t version;    // 파일 포맷 버전
uint64_t n_tensors;  // 텐서의 수
uint64_t n_kv;       // 키-값 쌍의 수
```

**② 메타데이터 (n_kv개의 키-값 쌍)** — 모델에 대한 정보

```c
struct gguf_kv {
    struct gguf_str key;     // 키(문자열)
    enum gguf_type type;     // 값의 타입
    union gguf_value value;  // 실제 값
};
```

**③ 텐서 정보 (n_tensors개)**

```c
struct gguf_tensor_info {
    struct gguf_str name;        // 텐서의 이름
    uint32_t n_dims;             // 차원 수
    uint64_t ne[GGML_MAX_DIMS];  // 각 차원의 크기
    enum ggml_type type;         // 텐서의 타입
    uint64_t offset;             // 데이터 섹션에서의 오프셋
};
```

**④ 텐서 데이터** — 마지막으로 실제 가중치 바이트가 들어간다.

핵심은 ③번 단계에서 **모든 텐서의 오프셋이 파일 안에 미리 기록**되어 있다는 점이다. 파일을 끝까지 파싱하지 않아도 특정 텐서가 어느 바이트 구간에 있는지 바로 계산할 수 있고, 이 구조가 5장에서 다루는 메모리 매핑의 전제가 된다.

---

## 5. GGUF가 제공하는 장점

### 5-1. 메모리 매핑 (mmap)

필요한 부분만 메모리에 올릴 수 있다는 것이 핵심이다.

**일반적인 로딩 방식**은 다음 순서를 거친다.

```
SSD
 ↓
파일 읽기
 ↓
RAM 전체 복사
 ↓
Tensor 생성
 ↓
GPU 또는 CPU로 복사
```

예를 들어 13GB 모델이라면, 실제로 쓰든 안 쓰든 아래처럼 전체를 RAM에 올려야 한다.

```
13GB 모델
SSD → RAM(13GB) → Tensor → GPU
```

**GGUF는** 텐서 오프셋이 헤더에 고정되어 있어서, 파일을 가상 메모리에 매핑(mmap)해두고 실제로 필요한 페이지만 그때그때 불러올 수 있다.

```
SSD
 │
 │ (Virtual Memory)
 ▼
Memory Mapping(mmap)
 ↓
Virtual Memory에 연결
 ↓
필요한 Page 접근
 ↓
Page Fault
 ↓
OS가 SSD에서 해당 Page만 RAM으로 가져온다.
```

즉 모델 전체를 미리 RAM에 복사하는 대신, 실제로 연산에 쓰이는 텐서 블록이 접근될 때마다 OS가 페이지 폴트를 통해 그 부분만 로드한다.

### 5-2. 양자화 적용

이미 양자화가 적용된 상태로 저장되기 때문에 메모리 사용량이 적고, 로드 후 별도 변환 없이 바로 연산에 쓰인다.

FP16과 Q4(4bit 양자화)로 같은 4096개 값을 저장한다고 비교하면 차이가 뚜렷하다.

| 타입 | 값 개수 | 크기 |
|---|:--:|:--:|
| FP16 | 4096개 | 8192 Byte |
| Q4 | 4096개 | 2048 Byte |

메모리에서 읽는 양 자체가 **4배** 줄어든다. LLM 추론은 대부분 **Memory Bound**(연산량보다 메모리 대역폭이 병목)라서, CPU에서는 연산 자체보다 메모리를 읽어오는 속도가 더 느린 경우가 많다. 그래서 양자화의 효과는 단순히 연산량이 줄어드는 것보다 **메모리 대역폭 절감**에서 훨씬 크게 나타난다.

### 5-3. Sequential(순차) 저장

앞서 본 것처럼 GGUF는 아래 순서로 저장되어 있다.

```
Header
Metadata
Tensor1
Tensor2
Tensor3
Tensor4
...
```

이 순차 저장 구조는 다음 흐름으로 이어진다.

```
Sequential Read
 ↓
OS Read Ahead
 ↓
CPU Prefetch
 ↓
Cache Hit 증가
```

SSD는 연속된 데이터를 읽는 처리 속도가 랜덤 접근보다 훨씬 빠르기 때문에, GGUF처럼 연속된 구조로 저장된 파일은 읽어들이는 속도도 함께 빨라진다. 결과적으로 **메모리 효율성과 로딩 속도** 두 측면에서 GGUF가 유리하다.

## 6. Llama.cpp 기반 로컬 추론 환경에서 GGUF를 쓰는 이유

GGUF는 `llama.cpp`를 포함한 GGML 기반 추론 엔진에서 효율적으로 쓰이도록 설계된 **추론 중심 모델 포맷**이다.

`llama.cpp`의 목표는 **GPU 없이도 LLaMA를 실행하는 것**이다.

- LLaMA: Meta에서 만든 대규모 언어 모델
- llama.cpp: LLaMA 모델을 CPU/GPU에서 돌리기 위한 추론 엔진
- 비유하면 LLaMA는 요리책, llama.cpp는 그 요리책을 보고 요리하는 요리사다

```
Meta
 │
 │ 학습
 ▼
Llama 3.1 8B (모델)
 │
 │ 저장
 ▼
GGUF
 │
 │ 실행
 ▼
llama.cpp
 │
 ▼
"안녕하세요" → "무엇을 도와드릴까요?"
```

llama를 어디서든 가볍게 사용하는 것이 목표이기 때문에, 기존 LLM 추론의 기본 전제였던

```
PyTorch
 ↓
GPU
 ↓
RTX4090
```

가 아니라

```
MacBook
노트북
CPU
라즈베리파이
```

에서도 돌아가야 한다. 그러려면 다음이 필요하다.

- 메모리를 적게 써야 하고
- 로딩이 빨라야 하고
- CPU 연산이 빨라야 한다.

GGUF는 변환하는 모델의 양자화된 가중치를 모두 저장해 이 요구를 충족한다.

**단일 파일이라는 이점도 있다.** HuggingFace에서 모델을 받으면 보통 아래처럼 여러 파일이 필요하다.

```
config.json
tokenizer.json
tokenizer.model
generation.json
model.safetensors
...
```

반면 GGUF는 `model.gguf` 하나면 된다. 이 파일 안에 Tokenizer, RoPE 설정, Context Length, Special Token, Architecture, Tensor, Vocabulary가 전부 들어 있기 때문이다. 파일 하나로 모델을 통째로 옮기고 실행할 수 있다는 점이 로컬 환경 배포·실행을 단순하게 만든다.

## 7. 모델 파일 구조 & 실행 환경 최적화 관점에서 본 GGUF의 장점

**① 모델 파일 구조 관점**

| 구조적 특징 | 제공하는 장점 |
|---|---|
| 헤더에 텐서 오프셋을 미리 기록 (4장) | 파일을 파싱하지 않고도 각 텐서 위치를 바로 계산 → mmap이 가능해짐 (5-1) |
| Header → Metadata → Tensor1, Tensor2, ... 순차 배치 (4장) | SSD의 순차 읽기 성능을 그대로 활용, OS Read Ahead·CPU Prefetch로 이어져 Cache Hit 증가 (5-3) |
| 메타데이터(Tokenizer, RoPE, Context Length 등)를 텐서와 한 파일에 통합 | `config.json`, `tokenizer.json` 등 여러 파일로 흩어지는 HuggingFace 방식과 달리 `model.gguf` 하나로 배포·실행 (6장) |
| 텐서 타입 자체에 양자화 정보 포함 | 별도 변환 단계 없이 파일을 열자마자 양자화된 가중치를 그대로 사용 |


**② 실행 환경 최적화 관점**

| 실행 환경 요구사항 | GGUF가 충족하는 방식 |
|---|---|
| 메모리를 적게 써야 한다 | 양자화가 파일에 내장되어 있어 FP16 대비 Q4는 읽는 양이 **4배** 줄어든다 (5-2) |
| 로딩이 빨라야 한다 | mmap + 순차 저장 구조로 필요한 페이지만, 그것도 빠르게 불러온다 (5-1, 5-3) |
| CPU 연산이 빨라야 한다 | LLM 추론은 **Memory Bound**라 대역폭 절감(양자화)이 연산량 절감보다 체감 성능에 더 크게 기여한다 (5-2) |
| GPU 없는 환경(MacBook, 라즈베리파이 등)에서도 돌아가야 한다 | 위 세 가지가 합쳐져 CPU 단독 환경에서도 실질적으로 실행 가능한 수준의 성능을 낸다 (6장) |


정리하면, **파일 구조 관점**에서 GGUF는 "오프셋을 미리 알 수 있는 단일 파일"이라는 설계로 mmap과 순차 읽기를 가능하게 만들고, **실행 환경 최적화 관점**에서는 그 구조 위에 내장 양자화를 얹어 메모리 대역폭이 병목인 CPU 환경에서도 로딩과 추론이 가볍게 돌아가도록 만든다. 이 두 관점이 맞물린 결과가 곧 `llama.cpp`가 GGUF를 표준 포맷으로 채택한 이유다.

## 정리

| 항목 | PyTorch(.pt/.safetensors) | GGUF |
|---|---|---|
| 목적 | 학습 + 추론 | 추론 전용 |
| 양자화 | 별도 수행 가능 | 파일에 포함 |
| 메타데이터 | 여러 파일에 분산 | 하나의 파일에 포함 |
| Memory Mapping | 제한적 활용 | mmap 기반 최적화 |
| CPU 추론 | 비효율적 | 최적화 |
| llama.cpp 지원 | 직접 사용 어려움 | 기본 지원 |

결국 GGUF는 **오프셋이 고정된 단일 파일 구조 + mmap + 순차 저장 + 내장 양자화**를 통해 메모리 효율성과 로딩 속도를 모두 잡은 포맷이며, 이 설계 목표가 "GPU 없이 가볍게 어디서든 LLM을 돌린다"는 `llama.cpp`의 목표와 정확히 맞아떨어지기 때문에 둘이 표준 조합으로 쓰인다.
