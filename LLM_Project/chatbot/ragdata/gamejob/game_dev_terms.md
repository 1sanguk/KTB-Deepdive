# 게임 개발 용어 및 프로세스

## 개발 단계 용어

### 프리프로덕션 (Pre-Production)

본격 개발 전 기획과 프로토타이핑 단계.

- **컨셉(Concept)**: 게임의 핵심 아이디어와 장르, 타겟 플랫폼, 분위기를 정의한다.
- **프로토타입(Prototype)**: 핵심 게임플레이 메카닉이 재미있는지 검증하기 위한 초기 실험 버전. 그래픽은 거칠어도 무방하다.
- **버티컬 슬라이스(Vertical Slice)**: 게임의 한 구간을 완성도 높게 구현한 샘플. 투자자나 퍼블리셔에게 게임의 방향성을 보여주기 위해 사용된다.
- **GDD(Game Design Document)**: 게임의 전체 기획을 문서화한 것. 시스템 규칙, 수치, UI 구조, 스토리, 플로우 차트를 포함한다.

### 프로덕션 (Production)

실제 게임 콘텐츠와 시스템을 개발하는 단계.

- **알파(Alpha)**: 핵심 기능은 완성되었으나 콘텐츠가 미완성이고 버그가 많은 단계. 내부 테스트를 시작한다.
- **베타(Beta)**: 대부분의 콘텐츠가 완성되고 외부 테스트(CBT/OBT)를 진행하는 단계.
  - **CBT(Closed Beta Test)**: 선별된 유저를 대상으로 한 비공개 테스트.
  - **OBT(Open Beta Test)**: 누구나 참여 가능한 공개 테스트.
- **골드 마스터(Gold Master / Gold)**: 출시 가능한 최종 빌드. QA를 통과하고 더 이상 크리티컬 버그가 없는 상태.
- **런칭(Launch)**: 게임 공식 출시.
- **소프트 런칭(Soft Launch)**: 특정 국가 또는 지역에만 먼저 출시해 데이터를 수집하고 문제를 수정한 뒤 글로벌 출시하는 전략.

### 포스트 프로덕션 (Post-Production / 라이브 서비스)

출시 이후 게임을 유지·운영하는 단계.

- **패치(Patch)**: 버그 수정 및 소규모 업데이트.
- **핫픽스(Hotfix)**: 긴급하게 배포되는 즉각적 버그 수정 패치.
- **메이저 업데이트**: 새 콘텐츠(신규 캐릭터, 던전, 시즌)를 추가하는 대규모 업데이트.
- **시즌(Season)**: 일정 기간 동안 운영되는 테마 콘텐츠 단위. 시즌 종료 후 보상을 지급하고 새 시즌을 시작한다.
- **서비스 종료(EOL: End of Life)**: 게임 서비스를 중단하는 것. "셧다운"이라고도 한다.

---

## 마일스톤 (Milestone)

프로젝트의 주요 완료 시점. PM이 일정을 관리할 때 사용하는 체크포인트.

- M0: 프리프로덕션 완료, 프로덕션 착수 승인
- M1: 알파 빌드 완료
- M2: 베타 빌드 완료 (CBT 가능 수준)
- M3: 골드 마스터 (출시)

## 게임 개발 기술 용어

### 게임 엔진

- **Unity**: C# 기반. 모바일·인디 게임에 많이 사용된다. 에셋스토어 생태계가 풍부하다.
- **Unreal Engine(언리얼 엔진)**: C++ + Blueprint 기반. AAA 게임, 콘솔 게임에 많이 사용된다. 그래픽 품질이 높다.
- **자체 엔진(In-house Engine)**: 넥슨의 NxGame, 엔씨소프트의 BladeEngine 등 대형 게임사는 자체 엔진을 사용하기도 한다.

### 렌더링 관련 용어

- **폴리곤(Polygon)**: 3D 모델을 구성하는 삼각형 면의 단위. 폴리곤 수가 많을수록 정밀하지만 성능 비용이 크다.
- **드로우콜(Draw Call)**: CPU가 GPU에게 오브젝트를 그리라고 명령하는 횟수. 드로우콜이 많으면 성능이 저하된다.
- **LOD(Level of Detail)**: 카메라와의 거리에 따라 오브젝트의 폴리곤 수를 단계적으로 낮추는 최적화 기법.
- **배칭(Batching)**: 여러 드로우콜을 하나로 묶어 GPU 부하를 줄이는 기법.
- **셰이더(Shader)**: GPU에서 실행되는 렌더링 프로그램. 버텍스 셰이더(위치 계산)와 프래그먼트 셰이더(색상 계산)로 나뉜다.
- **PBR(Physically Based Rendering)**: 물리 기반 렌더링. 빛과 물질의 실제 물리 법칙을 시뮬레이션해 사실적인 그래픽을 만든다.

### 게임플레이 시스템 용어

- **틱(Tick)**: 게임 루프가 한 번 업데이트되는 단위. 60FPS 게임은 1초에 60틱이다.
- **FPS(Frames Per Second)**: 초당 렌더링되는 프레임 수. 30fps는 부드러운 최소 기준, 60fps는 부드러운 경험, 120fps 이상은 고성능.
- **레이턴시(Latency)**: 클라이언트와 서버 간 통신 지연 시간. 핑(Ping, ms)으로 표현한다.
- **동기화(Sync)**: 멀티플레이어에서 모든 클라이언트의 게임 상태를 일치시키는 것. 권위 서버(Authoritative Server) 구조가 표준이다.
- **Dead Reckoning**: 네트워크 지연을 숨기기 위해 클라이언트가 서버 응답 전에 이동을 예측하는 기법.

### 빌드 및 배포

- **빌드(Build)**: 소스 코드를 컴파일해 실행 가능한 게임 파일로 만드는 과정.
- **CI/CD(지속적 통합/배포)**: 코드 변경 시 자동으로 빌드·테스트·배포하는 파이프라인. Jenkins, GitHub Actions 등을 사용한다.
- **스토어 심사**: 구글 플레이, 애플 앱스토어, 스팀 등에 제출하면 가이드라인 위반 여부 심사를 받는다.
- **패치 배포**: 클라이언트 패치는 스토어를 통해, 서버 패치는 서버 점검(정기/임시)으로 배포한다.

## 협업 도구

- **지라(Jira)**: 이슈 추적, 스프린트 관리, 칸반 보드. 게임 업계 표준 PM 툴.
- **컨플루언스(Confluence)**: 팀 위키 문서화 도구. GDD, 기술 문서를 공유한다.
- **퍼포스(Perforce / Helix Core)**: 대형 바이너리(아트 에셋, 레벨 파일) 버전 관리에 사용. AAA 게임 개발 표준.
- **Git / GitHub / GitLab**: 소스 코드 버전 관리. 소규모 팀에서 많이 사용한다.
- **슬랙(Slack) / 노션(Notion)**: 팀 커뮤니케이션 및 문서 관리.

## 게임 개발 방법론

- **워터폴(Waterfall)**: 기획 → 개발 → QA → 출시를 순서대로 진행. 대형 콘솔 프로젝트에 많이 사용된다.
- **애자일(Agile) / 스크럼(Scrum)**: 2주 단위 스프린트로 개발하며 빠르게 반복·개선한다. 라이브 서비스 게임 운영에 적합하다.
- **칸반(Kanban)**: 할 일 → 진행 중 → 완료 보드로 작업 흐름을 시각화한다.
- **크런치(Crunch)**: 출시 직전 야근과 초과 근무가 집중되는 시기. 게임 업계의 고질적 문제로 업계 내 개선 논의가 활발하다.
